### 1. 案件の概要とスコープ

| 項目             | 内容                                                                                                       | 決定事項                                               |
| :--------------- | :--------------------------------------------------------------------------------------------------------- | :----------------------------------------------------- |
| **システム**     | 塾向けの e ラーニングシステム。PC ブラウザからのアクセスを想定した Web システム。                          | -                                                      |
| **構築フェーズ** | 今回のスコープは**設計支援**のみ（構築はクライアント側で実施予定）。                                       | -                                                      |
| **利用地域**     | 日本国内のみを想定。                                                                                       | **東京リージョン**を使用。                             |
| **アクセス傾向** | 時間帯によりアクセスに大きな変動がある（学校時間帯は少なく、下校・帰宅後にピーク）。                       | コストを抑えつつ、ピーク時に耐えられる構成を検討。     |
| **予算感**       | 新規事業のため、具体的な予算は未定だが、極力抑えたい。ただし、性能不足によるユーザーの機会損失は避けたい。 | 利用状況に応じて柔軟にコストを最適化できる構成を採用。 |

### 2. AWS の主要構成要素（アドバイスと決定事項）

| 分野                 | 決定・推奨サービス                           | 理由・要件                                                                                                         |
| :------------------- | :------------------------------------------- | :----------------------------------------------------------------------------------------------------------------- |
| **スケーリング**     | **Auto Scaling**                             | アクセス変動に対応し、コストの最小化とピーク時の耐障害性を両立するため。                                           |
| **負荷分散**         | **Application Load Balancer (ALB)**          | Auto Scaling で増減するサーバーへリクエストを分散させるため。                                                      |
| **冗長性**           | **複数アベイラビリティゾーン (AZ)**          | サービスの推奨通り、システム全体の冗長性を確保するため。                                                           |
| **DNS/SSL**          | **Route 53** / **ACM (Certificate Manager)** | AWS の推奨サービスを利用。SSL 証明書は ACM で管理し、ALB までの HTTPS 通信を採用。ALB より内部は HTTP 通信とする。 |
| **静的コンテンツ**   | **S3**                                       | 画像などの教材データ保存先として。サーバー内ディスクと比べ、コストが安価で容量を気にせず利用できるため。           |
| **ネットワーク**     | **プライベートサブネット**                   | サーバー（EC2）は内部ネットワークであるプライベートサブネットに配置する。                                          |
| **外部通信**         | **NAT Gateway**                              | アプリケーションが外部 API を呼び出す（Outbound 通信）ために必要。冗長性を考慮し複数 AZ に配置。                   |
| **データベース**     | **RDS (Aurora 推奨)**                        | アプリケーション要件（MySQL/PostgreSQL 環境）を満たすフルマネージドサービス。運用負担を軽減。                      |
| **DB 構成**          | **マルチ AZ 構成**                           | 冗長性を確保。**読み込み（Reader）**と**書き込み（Writer）**のエンドポイントを分け、アプリ側で使い分けを実装する。 |
| **サーバーアクセス** | **Systems Manager (SSM)**                    | セキュアかつコスト効率の高いサーバーアクセス（SSH 踏み台サーバー不要）を実現するため。                             |
| **予備アクセス**     | **SSH**                                      | SSM に万一の障害が発生した場合の予備として、環境構築後に SSH 公開鍵を設定する方式を検討する。                      |
| **ログ管理**         | **S3**                                       | アプリケーションログは調査用に**S3**へアーカイブし、コストを最適化する。                                           |

### 3. 今後の流れとスケジュール感

1.  **クライアント側作業：** いたち様が、今回のヒアリング内容に基づき、**ALB、Auto Scaling、NAT Gateway、RDS（マルチ AZ）**などを含めた構成図の作成を試みる。
2.  **設計発注：** いたち様から構成図と合わせて発注書を送付（目安：約 5 営業日以内）。
3.  **契約手続き：** 発注書受領後、2〜3 営業日で契約締結。
4.  **案件開始：** 契約後、設計担当者への引き継ぎを行い、キックオフを実施。
5.  **設計納品：** キックオフから約 5〜7 営業日で設計書を納品予定。

### 4. 確認事項

- ドメイン名については、一旦仮置きで設計を進める。決まり次第、改めて猫田様側に連携する。
- RDS の構成（特に読み込み/書き込みのエンドポイントの使い分け）については、構成図作成およびアプリケーション開発会社との連携が必要となる。
